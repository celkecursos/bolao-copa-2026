# TASK 12 — Deploy (2/3): Publicar a aplicação na VPS

Segunda fase de deploy. Clona o projeto da branch `main` via SSH, instala dependências, cria o `.env` de **produção** e roda as migrations + seed de produção (jogos reais via API). Requer a **Task 11** concluída.

> **Credenciais:** seções **COMUM** + **PRODUÇÃO** (`PROD_*`) e **SERVIDOR/VPS** + **GIT** do `CREDENCIAIS.txt`.
> ⚠️ **Use SOMENTE as credenciais de PRODUÇÃO (`PROD_*`).** No `.env` da VPS as chaves vão **sem o prefixo** (ex.: `PROD_DB_PASSWORD` → `DB_PASSWORD`).
>
> ⚠️ **Senhas com `#`:** no `.env`, valores com `#` devem estar entre aspas duplas, caso contrário o `#` é tratado como comentário e a senha fica truncada. Exemplo: `DB_PASSWORD="<senha_com_cerquilha>"`.

---

## 1. Clonar o projeto (branch main via SSH)

Acessar a VPS e ir ao diretório do projeto (use `VPS_SSH_USER`, `VPS_IP` e `VPS_PROJECT_PATH` do `CREDENCIAIS.txt`):
```
ssh <VPS_SSH_USER>@<VPS_IP>
cd <VPS_PROJECT_PATH>
```

Se o diretório já tiver arquivos de um teste anterior, limpar (cuidado — apaga tudo):
```
rm -rf <VPS_PROJECT_PATH>/{*,.*} 2>/dev/null; ls -la
```

Clonar a branch de produção usando `DEPLOY_BRANCH` e `REPOSITORIO_GIT_SSH` do `CREDENCIAIS.txt`:
```
git clone -b <DEPLOY_BRANCH> <REPOSITORIO_GIT_SSH> .
```

Marcar o diretório como seguro para o Git (use `VPS_PROJECT_PATH` do `CREDENCIAIS.txt`):
```
git config --global --add safe.directory <VPS_PROJECT_PATH>
```

---

## 2. Criar o `.env` de produção

```
cp .env.example .env
nano .env
```

Preencher a partir do `CREDENCIAIS.txt` (**COMUM + PRODUÇÃO**). Valores-chave:
```
APP_NAME="Bolão Copa 2026"
APP_ENV=production
APP_KEY=
APP_DEBUG=false
APP_TIMEZONE=UTC
APP_URL=<PROD_APP_URL>

DB_CONNECTION=mysql
DB_HOST=127.0.0.1
DB_PORT=3306
DB_DATABASE=copa
DB_USERNAME=copa
DB_PASSWORD="<PROD_DB_PASSWORD>"

SESSION_DRIVER=database
QUEUE_CONNECTION=database

# Super-admin criado pelo seeder (RolesAndPermissionsSeeder)
SEED_SUPERADMIN_NAME="Super Admin"
SEED_SUPERADMIN_EMAIL=<PROD_SEED_SUPERADMIN_EMAIL>
SEED_SUPERADMIN_PASSWORD="<PROD_SEED_SUPERADMIN_PASSWORD>"

# API de resultados — football-data.org
FOOTBALL_DATA_TOKEN=<FOOTBALL_DATA_TOKEN>

# E-mail iAgente (produção) — use MAIL_SCHEME, não MAIL_ENCRYPTION (Laravel 11+)
MAIL_MAILER=smtp
MAIL_SCHEME=ssl
MAIL_HOST=<PROD_MAIL_HOST>
MAIL_PORT=587
MAIL_USERNAME=<PROD_MAIL_USERNAME>
MAIL_PASSWORD="<PROD_MAIL_PASSWORD>"
MAIL_FROM_ADDRESS=<PROD_MAIL_FROM_ADDRESS>
MAIL_FROM_NAME="Bolão Copa 2026"
```

> Salvar no nano: `Ctrl + O`, Enter; sair: `Ctrl + X`.
>
> ⚠️ **Laravel 11+** usa `MAIL_SCHEME=ssl` em vez de `MAIL_ENCRYPTION=ssl`. A chave `MAIL_ENCRYPTION` não tem efeito nesta versão.

## 3. Instalar dependências e gerar a build

Dependências do PHP (otimizado para produção):
```
composer install --no-dev --optimize-autoloader
```

Dependências do Node.js e build de produção (**não** use `npm run dev` no servidor):
```
npm install
npm run build
```

---

## 4. Banco de dados, chave e permissões

Gerar a chave da aplicação:
```
/usr/bin/php8.3 artisan key:generate
```

Executar as migrations:
```
/usr/bin/php8.3 artisan migrate --force
```

Executar o **seed de produção** (importa jogos reais da API + cria palpites demo):
```
/usr/bin/php8.3 artisan db:seed --class=ProductionSeeder --force
```

> O `ProductionSeeder` executa em sequência:
> 1. `RolesAndPermissionsSeeder` — roles, permissões e super-admin (via `SEED_SUPERADMIN_*` do `.env`)
> 2. `worldcup:sync-matches` — importa times e jogos reais da API football-data.org
> 3. `DemoBetsSeeder` — 12 usuários demo com palpites nos jogos reais
>
> **Não use `WorldCup2026Seeder` em produção** — ele insere jogos fictícios. Em desenvolvimento use `php artisan migrate:fresh --seed` (chama `DatabaseSeeder`, que inclui o seeder fictício).

Ajustar a propriedade do diretório (use `VPS_LINUX_USER` e `VPS_PROJECT_PATH` do `CREDENCIAIS.txt`):
```
chown -R <VPS_LINUX_USER>:<VPS_LINUX_USER> <VPS_PROJECT_PATH>
chmod -R 775 storage bootstrap/cache
```

## 5. Cache de produção e Nginx

```
/usr/bin/php8.3 artisan config:cache
/usr/bin/php8.3 artisan route:cache
/usr/bin/php8.3 artisan view:cache
systemctl restart nginx
```

Acessar e validar: `<PROD_APP_URL>`

## Observações de troubleshooting

**MySQL: Access denied para o usuário da aplicação**
Se ocorrer `Access denied for user 'copa'@'localhost'` mesmo com `DB_HOST=127.0.0.1`, o PHP pode estar usando socket Unix em vez de TCP. Solução: adicionar `skip-name-resolve` ao MySQL e reiniciar:
```
echo '[mysqld]
skip-name-resolve' >> /etc/mysql/mysql.conf.d/mysqld.cnf
systemctl restart mysql
```

**Token da API inválido**
Se `worldcup:sync-matches` retornar "Nenhuma partida retornada", o token pode estar inválido ou expirado. Verifique em football-data.org → My Account → API Token, atualize no `.env` e limpe o cache:
```
/usr/bin/php8.3 artisan config:clear
/usr/bin/php8.3 artisan worldcup:sync-matches
/usr/bin/php8.3 artisan config:cache
```
