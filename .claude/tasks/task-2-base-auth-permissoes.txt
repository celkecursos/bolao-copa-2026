## 0. Credenciais e configuração (CONFIGURE AGORA, no `.env`)

> **Fonte das credenciais:** `CREDENCIAIS.txt` (na raiz), seção **DESENVOLVIMENTO**. Para **esta** task, basta ter no `.env` local o **necessário para rodar**: **banco de dados (`DEV_DB_*`)** e **super-admin (`DEV_SEED_SUPERADMIN_*`)**. Os demais valores (token da API, e-mail iAgente, URL do repo) podem ser preenchidos pelo usuário **enquanto as Tasks 1–7 rodam** — serão consolidados e validados na **Task 8 (Configurar .env)**, antes do commit. As chaves `DEV_*` mapeiam para o `.env` **sem o prefixo** (ex.: `DEV_DB_HOST` → `DB_HOST`). Registre as chaves no `.env.example` **sem os valores reais**.

**Nunca** versione o `.env`, o `CREDENCIAIS.txt` nem estes segredos em repositório público — se algum token vazar, gere um novo.

### API de resultados — football-data.org (usada na Task 7)
- Header de autenticação: **`X-Auth-Token`**
- O token (`FOOTBALL_DATA_TOKEN`) está no **`CREDENCIAIS.txt`** — não repita o valor aqui.
- ⚠️ **Throttling automático**: a API informa o limite de requisições nos **headers da resposta** (ex.: `X-Requests-Available-Minute`). O cliente **deve inspecionar esses headers** e respeitar o limite (free tier ~10 req/min). (Orientação do próprio provedor.)

### Envio de e-mail — servidor iAgente (usado na Task 7)
- Os e-mails do sistema serão enviados pelo **servidor da iAgente**.
- Deixe a config de e-mail **preparada para receber os dados da iAgente** (SMTP):
  ```
  MAIL_MAILER=smtp
  MAIL_HOST=<HOST_IAGENTE>
  MAIL_PORT=<PORTA_IAGENTE>
  MAIL_USERNAME=<USUARIO_IAGENTE>
  MAIL_PASSWORD=<SENHA_IAGENTE>
  MAIL_ENCRYPTION=<tls|ssl>
  MAIL_FROM_ADDRESS=<remetente>
  MAIL_FROM_NAME="Bolão Copa 2026"
  ```
- Enquanto os dados da iAgente não chegarem, use `MAIL_MAILER=log`.

## 1. Setup inicial (sobre o projeto criado na Task 1)

> O projeto Laravel **já foi criado na Task 1** (com MySQL, Pest, sem starter kit). Aqui adicionamos auth, front-end e fila.

- Confirme **Laravel 13**, **PHP 8.3** e **Node.js 24** (as MESMAS versões serão instaladas na VPS no deploy).
- **Tailwind CSS** (via Vite, config oficial do Laravel) para todo o front-end. Layout limpo, responsivo. Crie **componentes Blade reutilizáveis** e um **layout base** (header/nav/footer) que as demais telas usarão.
- **Tema claro/escuro com alternância**: configure `darkMode: 'class'` no `tailwind.config.js`; um botão de toggle (componente `<x-theme-toggle/>`) alterna a classe `dark` no `<html>` e **persiste em `localStorage`**; um script no `<head>` aplica o tema salvo **antes da renderização** (sem flash) e respeita a preferência do sistema. Disponível no app e nas telas de login (guest).
- **Fila**: configure `QUEUE_CONNECTION=database` (será usada nas Tasks 6 e 7) e crie a tabela de jobs.
- `README.md` inicial: como instalar (`composer install`, `npm install`), configurar `.env`, migrar e rodar (`php artisan serve` + `npm run dev` + `php artisan queue:work`).

## 3. Sistema de permissões (spatie/laravel-permission)

Crie 3 **papéis (roles)**:

1. **super-admin** — acesso total; passa por cima de qualquer verificação de permissão (use `Gate::before`).
2. **admin** — pode **cadastrar/editar times**, **cadastrar/editar jogos** e **atribuir/alterar o resultado** dos jogos.
3. **user** (usuário comum) — pode **fazer palpites** e ver o ranking; **não** edita times nem lança resultados.

Requisitos:

- **Seeder** que registra roles e permissions e cria um **super-admin** padrão (ex.: `superadmin@bolao.test` / senha definida no seeder e documentada no README).
- Middleware de role/permission nas rotas (`role:admin|super-admin`, etc.).
- Camada de **Policies** (`GamePolicy`, `BetPolicy`, `TeamPolicy`) + `@can` nas views para esconder ações sem permissão.
- Tela (apenas super-admin) para **listar usuários e atribuir/remover papéis**.

Permissions sugeridas: `teams.manage`, `games.manage`, `games.set-result`, `bets.create`, `users.manage`, `roles.manage`.
